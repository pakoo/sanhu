## 一、目标与范围

**核心痛点**：用户持有 8 只基金，但对"基金里到底买了什么"完全模糊。v1.5 的全部工作都围绕这个痛点展开。

**范围边界**：
- 权益类基金（股票型、混合型、QDII）→ 完整持仓穿透
- 债券型基金 → 只展示股/债/现金资产配比，不拆解债券明细
- 数据来源：Eastmoney fundf10 季报 API（每季度披露一次）
- 不做实时行情、不做个股深度分析

## 二、功能模块拆解

### 模块 T1 — 数据库迁移
新增两张表，向后兼容，不破坏现有数据。

```sql
-- 基金股票持仓（季报）
CREATE TABLE IF NOT EXISTS fund_holdings (
    code        TEXT NOT NULL,
    stock_code  TEXT NOT NULL,
    stock_name  TEXT NOT NULL,
    weight      REAL,            -- 占基金净值比例（%）
    industry    TEXT,
    report_date TEXT NOT NULL,   -- 格式：2024-09-30
    PRIMARY KEY (code, stock_code, report_date)
);

-- 基金行业分布（季报）
CREATE TABLE IF NOT EXISTS fund_industry (
    code        TEXT NOT NULL,
    industry    TEXT NOT NULL,
    weight      REAL,
    report_date TEXT NOT NULL,
    PRIMARY KEY (code, industry, report_date)
);
```

验收标准：
- `python -c "from backend.database import init_db; init_db()"` 无报错
- 两张表在 jijin.db 中存在，不影响现有表

### 模块 T2 — 持仓抓取（fetcher.py 扩展）
在 `backend/fetcher.py` 新增两个函数，不改动现有代码。

**函数 1：`fetch_fund_holdings(code: str, save: bool = True) -> list[dict]`**
- URL：`https://fundf10.eastmoney.com/FundArchivesDatas.aspx?type=jjcc&code={code}&topline=10`
- 返回格式：`[{"stock_code": "600519", "stock_name": "贵州茅台", "weight": 9.5, "industry": "食品饮料", "report_date": "2024-09-30"}, ...]`
- 解析方式：响应为 JSONP，正则提取后解析 `Data.fundStocks` 列表
- 债券基金（category=bond）跳过，直接返回 `[]`
- 存入 fund_holdings 表（INSERT OR REPLACE）

**函数 2：`fetch_fund_industry(code: str, save: bool = True) -> list[dict]`**
- URL：`https://fundf10.eastmoney.com/FundArchivesDatas.aspx?type=hycc&code={code}`
- 返回格式：`[{"industry": "科技", "weight": 35.2, "report_date": "2024-09-30"}, ...]`
- 解析方式：同上，解析 `Data.industryAlloc` 列表

**辅助函数：`refresh_holdings_for_all_funds() -> dict`**
- 遍历 holdings 表中所有 equity/mixed/qdii 基金，依次调用上两个函数
- 返回 `{"refreshed": [...], "skipped": [...], "errors": [...]}`

验收标准：
- 对 `006195`（国金量化）调用后，fund_holdings 中有数据，weight 总和 ≤ 100
- 对 `007171`（债基）调用返回 `[]`，不写入数据库

### 模块 T3 — 重叠分析计算（新建 backend/holdings_analysis.py）
核心业务逻辑文件，无网络请求，只读数据库。

**函数 1：`get_holdings_overlap() -> dict`**
逻辑：
1. 读取所有权益类基金最新一期的 fund_holdings 记录
2. 合并同一 stock_code 的多条记录，计算实质暴露：
   - 实质暴露 = Σ(基金对应 holdings 中的持仓金额占比 × 该基金中该股票的 weight)
3. 标出在 ≥2 只基金中出现的股票（重叠股票）
4. 计算 Herfindahl 指数：H = Σ(每只基金的实质暴露占总暴露%)²
5. 等效独立基金数 = 1 / H

返回结构：
```python
{
    "overlap_stocks": [
        {
            "stock_code": "300750",
            "stock_name": "宁德时代",
            "appear_in": ["006195", "021095", "024203"],  # 基金代码列表
            "total_exposure_pct": 2.3,                    # 对总资产的实质暴露%
        },
        ...
    ],
    "diversification_score": 2.3,   # 等效独立基金数
    "total_funds_analyzed": 5,
    "report_date_range": "2024-09-30",
}
```

**函数 2：`get_industry_breakdown() -> dict`**
逻辑：加权汇总所有权益基金的行业分布，返回穿透后的行业饼图数据

```python
{
    "industries": [
        {"industry": "科技", "weight": 35.2, "funds": ["006195", "021095"]},
        ...
    ],
    "report_date": "2024-09-30",
}
```

**函数 3：`get_holdings_changes(code: str) -> dict`**
逻辑：取该基金最近两期 report_date 的持仓对比

```python
{
    "new_stocks": [...],    # 本期有、上期没有
    "removed_stocks": [...], # 上期有、本期没有
    "increased": [...],      # weight 变化 > +1%
    "decreased": [...],      # weight 变化 < -1%
    "current_date": "2024-09-30",
    "prev_date": "2024-06-30",
}
```

验收标准：
- 单元测试：构造假数据，验证 Herfindahl 计算正确（2只各50% → H=0.5 → score=2）
- 集成测试：seed 数据后调用 `get_holdings_overlap()`，返回非空

### 模块 T4 — API 端点（app.py 扩展）
在现有 `app.py` 新增以下端点，不改动现有路由。

```
GET  /api/holdings/{code}/stocks      # 单基金持仓股票列表
GET  /api/holdings/{code}/industry    # 单基金行业分布
GET  /api/holdings/overlap            # 跨基金重叠分析
GET  /api/holdings/industry-total     # 穿透后汇总行业分布
GET  /api/holdings/{code}/changes     # 持仓变化对比
POST /api/holdings/refresh            # 手动触发抓取所有基金持仓
```

APScheduler 调度（在 app.py startup 事件中注册）：
```
每日 16:00（周一至周五）→ refresh_holdings_for_all_funds()
```

验收标准：
- `curl http://localhost:8000/api/holdings/006195/stocks` 返回正确 JSON
- `curl http://localhost:8000/api/holdings/overlap` 返回 overlap_stocks 列表
